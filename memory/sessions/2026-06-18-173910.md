## What happened

A real-use consideration the dogfood masked: my dogfood reviewer was a **spawned subagent with
filesystem access**, so "read these files" worked. But the common real-world reviewer is a
**web-based chat AI with no repo access** — there, cited paths don't resolve and the human must
manually attach files. Folded a fix into `second-opinion` (no version bump — v4.10.0 still uncommitted):

- **Step 3** now requires the snapshot to **stand alone** — put decision-relevant substance inline,
  not bare path citations a repo-less reviewer can't open.
- **Step 5 (hand off)** now distinguishes a **repo-access reviewer** (reads cited paths) from a
  **repo-less reviewer** (human pastes snapshot + attaches the named files).
- **Snapshot template** gains an **"Attach to reviewer (if it can't read the repo)"** section —
  the exact files/excerpts the human should bring (empty if self-contained).
- Recorded the refinement in `docs/DESIGN-fresh-context-review.md` §5a and the `fresh-review-v4100`
  fact.

No description change → adapters unchanged. memory-lint clean.

## Decisions Made

- The fresh-context snapshot must be **self-contained and name its attachments**, so review works
  for any AI (web chat included), not just a reviewer with filesystem access. Matches the tool's
  vendor-neutral ethos and the security advisory's "you are exporting to another system" framing.
