## What happened

- **Finding 1 — NL didn't fire the native Gemini command.** After `sync skill adapters` created
  `.gemini/commands/hello-world.toml`, "run hello-world" still read `agent-skills/` first; the
  user had to ask explicitly. **Diagnosed (confirmed against Gemini CLI docs):** Gemini custom
  commands are **slash commands** (`/<name>`, explicit) — not natural-language auto-triggers like
  Claude/Cursor/Kiro. So NL correctly routes through the `AGENTS.md` baseline to the **same**
  neutral skill. Not a defect: the slash command and the baseline are both pointers to
  `agent-skills/hello-world/SKILL.md`, so they can't diverge — `agent-skills/` is where both
  roads lead, not a detour.
- **Finding 2 — agent told the user to commit vendor dirs.** After sync, Gemini recommended
  committing `.gemini/` (and other adapter dirs). **Wrong:** adapters are gitignored, per-machine,
  regenerated; only `agent-skills/` is committed. The `.gitignore` already blocks them, but the
  agent should never *recommend* it. Root cause: `SKILLS.md` said adapters are gitignored but never
  explicitly said "don't commit / recommend committing."
- **Fixes (PATCH, no shape change):**
  - `SKILLS.md`: Gemini adapter documented as a **slash command `/<name>`**; a "trigger semantics
    differ per vendor" note; a **never-commit-the-adapters** guard on the `sync` operation
    (report adapters as "gitignored — do not commit; only `agent-skills/` is shared").
  - `AGENTS.md` (root + template): adapter line now says **"never commit them."**
  - `docs/DESIGN-skills-layer.md`: recipe table flags Gemini slash-invoked; trigger-semantics +
    commit-guard note; status line.
  - `VERSION`→4.5.1; `UPGRADE.md` 4.5.0→4.5.1 rung + table; `README`/`CHANGELOG`.
    `DECAY.md`/`REVIEW.md` unchanged.
