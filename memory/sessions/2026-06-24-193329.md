**Trigger (Copilot fresh-clone dogfood).** The maintainer removed simple-proxy + re-cloned fresh and
test-drove Copilot. Findings:
- **Memory bootstrap self-initializes:** plain `start` made Copilot read `AGENTS.md` + memory on its own
  (so "start from AGENTS.md" is unnecessary). ✅
- **But adapters + hook do NOT self-initialize on a clone:** the gitignored skill adapters are absent, and
  the git hook is unactivated (`core.hooksPath` is local config; git can't auto-run committed hooks on
  clone — security). The user had to run `sync skill adapters`, and (I verified on their clone) the hook
  was still **unactivated** — so the v4.19.0 auto-stub wouldn't have fired. Two easily-missed manual steps.
- **Strong validation (same dogfood):** Copilot praised clear separation of concerns ("deleting/recreating
  felt entirely safe — only Markdown + Bash"), idempotent syncing, "executable documentation", resiliency
  — "highly robust, deterministic, and easy to reason about."

**Fix.** `.githooks/init.sh` — one idempotent command that (1) regenerates the vendor skill adapters and
(2) runs `git config core.hooksPath .githooks`. It lives in `.githooks/` but is **not** a git-hook name,
so git never auto-runs it. Plus an **`AGENTS.md` self-init note** (root + template): on a first session in
a clone (adapters absent / `core.hooksPath` unset), run `bash .githooks/init.sh` — the agent does it
proactively (Copilot already reads AGENTS.md on `start`). So the fresh-clone setup is **one agent step**
(or one human command) instead of two; CI remains the zero-config floor regardless.
