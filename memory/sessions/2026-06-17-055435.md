## What happened

- **Screenshot.** User invoked **`/hello-world`** (Gemini slash command = native adapter). The skill
  ran (`hello.sh "Gemini CLI"` → greeting + local/UTC time + reminder). The agent reported
  *"Invocation Path: AGENTS.md baseline (Natural language request)"* — wrong again; and notably it
  did **not** do the folder scans this time, yet still mis-attributed.
- **Root cause (definitive).** A Gemini `/<name>` is **expanded into the adapter's `prompt` text
  before the model sees anything** — the model never receives the slash token, only the expanded
  prompt (indistinguishable from a typed request); the `SKILL.md` is read *after* the trigger. So an
  agent **structurally cannot** tell native-adapter from baseline, and **no `SKILL.md` prose can fix
  it** — both prior fixes (v4.5.1 "attribute by entry point", then "attribute from invocation, not a
  dir scan") were doomed. The `description` field only feeds `/help`; it isn't sent on invocation.
- **Decision — option B (drop the self-report).** The maintainer chose the lightweight path over
  (A) self-identifying adapters or (C) document-only.
- **Changes.**
  - `agent-skills/hello-world/SKILL.md` step 3 → **report the result, not the trigger**: confirm the
    skill ran + which neutral source was read; explicitly *do not* guess the invocation path (and a
    note on *why*, so it isn't re-added).
  - `docs/DESIGN-skills-layer.md` §4c → captured the root cause + the "B" decision + "if deterministic
    path proof is ever needed, carry it in the adapter's prompt text, not agent introspection."
- **Scope.** Neither `agent-skills/hello-world` nor the DESIGN doc is installed into targets, so
  **no VERSION bump** (consistent with prior demo-content edits).
