## Summary

Shipped **v4.22.3 (PATCH)** — retuned the post-commit active-session window from **2h → 30 min**, on
maintainer feedback that 2h is too long for the observed minutes-apart bursts.

**Why:** v4.22.1 used 2h (taken from the original reporter's suggestion). But the real pattern is
follow-up stubs **minutes** apart, and 2h is long enough to wrongly conflate a *genuinely new* working
session that begins within 2h of the previous session's log — the hook would nudge "enrich the old log"
for new work (false-suppression). **30 min** spans a session's commit cadence (incl. a short test/think
gap) while a new session after a longer pause still gets its own stub. Erring shorter is safer: an extra
mid-session stub (cleaned up by the review + the v4.22.2 agent rule) is milder than mislabeling a new
session.

**Unit change:** the override env var is now **`AGENT_MEMORY_SESSION_WINDOW_MINUTES`** (integer minutes),
replacing `_HOURS`. Necessary because BSD `date -v` rejects a fractional hour (`-v-0.5H`), so a sub-hour
default needs integer minutes. Both runtimes still supported: `date -v-30M` (BSD) / `date -d "30 minutes
ago"` (GNU).

**Validated:** isolated threshold cases — 2 min & 25 min → SUPPRESS; 45 min & no-files → CREATE; custom
`win=60` honored. Hook `bash -n` syntax OK.
