**Why.** Real client-team pain (relayed by the maintainer): the after-session ritual **isn't followed
through even with Claude** (busy sessions just end), and a **Copilot-only team has no triggers** for any
ritual. Root cause: rituals were agent-self-triggered conventions; the hook layer was opt-in, per-vendor,
and didn't travel into targets. The maintainer reaffirmed the Vision (lightweight / no-code /
agent-runtime) but said triggers + the end-of-session log should be reconsidered, **endorsed the
git-hook + CI direction**, and added a **zero-manual / untrained-user adoption constraint** ("any manual
operation is a barrier" as the protocol gains traction).

**Built.**
- **`.githooks/post-commit`** (bash, advisory, never blocks; bash 3.2-compatible): after a commit that did
  real work but carried no session log, **auto-stubs** one (`memory/sessions/<ts>.md` with the commit
  subject + `git diff --stat`) and nudges — the **capture/judgment split** (deterministic capture; the
  thoughtful summary stays the agent's job, like `memory-lint`). Also re-syncs adapters when a skill
  changed. Skips merges; doesn't pile up stubs. + `.githooks/README.md`.
- **`.github/workflows/agent-memory.yml`** — the **CI floor**: runs `memory-lint` + an advisory
  session-log presence check on push/PR with **zero per-user setup** (the untrained-user-proof layer).
  Advisory by default; opt-in gate `AGENT_MEMORY_STRICT=1`.
- **Activation is agent-run, no manual user step:** ENABLE installs both + runs
  `git config core.hooksPath .githooks`. **Honest limit recorded:** git can't auto-run committed hooks on
  a fresh clone (security) → **CI is the always-on backstop**; the local hook is agent-activated.
