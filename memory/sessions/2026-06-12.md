## Session 2

**Agent:** Claude Code
**User:** First real-world test drive — AI-enable `~/sandbox/mercury-composable`
(a real Java/Maven repo) and surface design improvements.
**Duration:** medium

### What We Did

Ran ENABLE.md end-to-end against mercury-composable. Detection → Mode C (existing
hand-written, non-ours `CLAUDE.md`; no in-repo history). User chose an "improved"
integration variant over the stock protocol. Moved original to
`legacy/claude-code/CLAUDE.md` via `git mv`, integrated its content into
`memory/instructions.md` schema sections, generated `continuity.md`, installed all
6 bootstraps + `.agent/schema.md`, and appended the AI block to `.gitignore`.
First time the protocol was executed for real (prior examples were mockups).

### Design findings (the point of the exercise)

1. **Stock MIGRATE.md downgrades good steering files.** It dumps an existing
   `CLAUDE.md` verbatim under `## Migrated rules from <vendor>` and demotes the
   root file to a stub. When the vendor file already IS good project instructions,
   it should be INTEGRATED into the schema sections. Need an "integration" path.
2. **`legacy/` is gitignored by Step 7 — contradicts "originals preserved".**
   README/MIGRATE promise preserved originals, but ENABLE Step 7 adds `legacy/` to
   `.gitignore`, so after `git add . && commit` they are excluded from git. The
   preservation is local-only; teammates cloning never see them. Real contradiction.
3. **No source-of-truth / version-drift handling.** mercury had 3 different version
   strings (pom 4.4.11, old CLAUDE 4.4.3, README 4.3.69). Protocol gives no guidance;
   should say "prefer build manifest, surface drift as Open Thread."
4. **Protocol isn't git-aware.** MIGRATE says "move (do not copy)"; for tracked
   files `git mv` preserves history and should be specified.
5. **No verification/Step-after-report.** Nothing validates output; "confirm
   integration is faithful" had to be added as a manual Open Thread in the target.
6. **Templates are single-project shaped.** Real repo was a multi-module Maven
   reactor; schema absorbed it but there's no monorepo guidance.

### Decisions Made

- Adopted "improved integration" as the preferred behavior for good steering files
  (validated on a real repo). Candidate to fold into MIGRATE.md.

### Context for Next Session

Target repo (mercury-composable) is enabled; user is switching there to continue.
Tool-side improvements above are NOT yet implemented in ENABLE.md/MIGRATE.md —
see new Open Threads in continuity.md. Findings #1 and #2 are the highest-value.

## Session 7

**Agent:** Claude Code
**User:** Solve the multi-contributor merge conflict problem on session files.

### What We Did

Changed the session filename schema from `YYYY-MM-DD.md` (one file per day,
shared) to `YYYY-MM-DD-XXXX.md` (one file per session, with a 4-char random hex
suffix generated at session-start). Each contributor creates their own file;
no two contributors ever write to the same path on the same day, eliminating the
merge conflict entirely.

Updated everywhere the old format was specified:
- `templates/.agent/schema.md` — new filename format + rule "never append to
  another contributor's session file"; updated INDEX.md line format to match
- `templates/AGENTS.md` — After Every Session step 1: create, not append
- `AGENTS.md` (this repo's own protocol) — same
- `MIGRATE.md` — Principle 3, Section C workflow, JSONL reference
- `README.md` — migration description
- `memory/continuity.md` — Key Decisions reference
- `examples/migrated-cursor-aider-project/` — renamed the 3 session files
  and updated `ENABLE_OUTPUT.md` to show the new filenames

The `rust-event-bus` example has no real session files (only `.gitkeep`) so
nothing to rename there.

### Decisions Made

- One file per session (not per day) — the suffix is the discriminator, not a
  subdirectory or username, so no per-user config is needed.
- 4 hex characters: ~65 k possibilities, negligible collision risk for typical
  team sizes.
- `continuity.md`'s `last_session` field stays as-is — it's a best-effort
  quick-glance field; the one-line merge conflict it causes is trivially resolved.
- Existing committed session files (this repo's own `2026-06-12.md`) are left
  as-is; the schema change is forward-looking only.

### Context for Next Session

Session filename schema is now multi-contributor safe. Remaining open design
threads: source-of-truth/version-drift guidance, git-aware migration, post-report
verification step, monorepo guidance.

### What We Did

Changed the session filename schema from `YYYY-MM-DD-XXXX.md` (random hex) to
`YYYY-MM-DD-HHMMSS.md` (UTC timestamp, colons omitted for cross-platform
compatibility). Updated all references across the tool and examples:

- `templates/.agent/schema.md` — new filename format; reading protocol now says
  "sort lexicographically, read the most recent 2–3 files"; updated INDEX.md
  line format; simplified session file header from `# Sessions — YYYY-MM-DD` +
  `## Session N` to `# Session — YYYY-MM-DD-HHMMSS UTC` (one session per file,
  no numbering needed)
- `templates/AGENTS.md`, `AGENTS.md` — After Every Session step 1 updated
- `MIGRATE.md` — Principle 3, Section C workflow, JSONL reference; migration
  now uses original session timestamp (UTC) for the filename
- `README.md`, `memory/continuity.md` — format references updated
- `examples/migrated-cursor-aider-project/` — renamed 3 session files to
  `*-100000.md` and updated their internal headers and section levels to match
  the new one-session-per-file schema; updated `ENABLE_OUTPUT.md`

### Decisions Made

- UTC timestamp in the filename is the canonical session identifier — no config,
  no randomness, unambiguous sort order across any number of contributors.
- Colons omitted (`053000` not `05:30:00`) for Windows/FAT filesystem compatibility.
- Session file header simplified to `# Session — YYYY-MM-DD-HHMMSS UTC`; the
  `## Session N` numbering is dropped since each file holds exactly one session.
- `continuity.md`'s `last_session` field kept as-is (best-effort; trivial to
  resolve on merge).

### Context for Next Session
