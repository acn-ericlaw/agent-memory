Shipped **v4.22.1 (PATCH)** — fixed the `.githooks/post-commit` auto-stub to fire **per working

(`review-scratch/feedback-2026-06-27-post-commit-session-stub.md`).

**Bug:** the auto-stub's only de-dup guard checked for an *untracked* stub. Once the agent enriched +
**committed** the session log, the next work commit found no waiting stub → wrote a **fresh** one. A
multi-commit session produced ~6 near-identical lite logs (each with an extra `memory:` commit). Worse:
the decay model counts session files, so a session landing N commits counted as N sessions →
`sessions_since_last_used` inflated → facts decayed ~N× too fast (ironic, given `memory-lint` exists to
catch decay miscounts).

**Fix (`.githooks/post-commit`):** suppress a new stub when a session log already exists **within an
active-session window** (default 2h; override `AGENT_MEMORY_SESSION_WINDOW_HOURS`); nudge the agent to
**enrich the existing log** instead. Detection is by the newest session **filename** timestamp —
deliberately **not `mtime`** (a `git clone`/checkout resets mtime to now, which would wrongly suppress
stubs for hours after a clone). Compared lexicographically to a window-ago stamp (`YYYY-MM-DD-HHMMSS`
sorts chronologically). This **subsumes** the old untracked guard (a fresh stub has a recent filename)
and covers the just-committed case. A genuinely new session (no log within the window) still gets a stub —
the "no silent gap" guarantee holds. Bash 3.2-compatible: BSD `date -v-2H` with a GNU `date -d "2 hours
ago"` fallback; if both fail, it falls through to stubbing (safe default).

- End-to-end in a temp git repo with the real hook: S1 work commit → 1 stub; S2 commit the stub; S3
  second work commit → **SUPPRESS** (nudge to enrich), still 1 stub (the bug would make it 2). ✓
