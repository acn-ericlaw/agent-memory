## What happened

**Design-altitude work (no code yet):** drafted `docs/DESIGN-ritual-triggers.md` (PROPOSED) and opened
the `(blueprint)` gap **`bp-ritual-triggers`** → serves `vision-agent-memory`.

**Trigger.** Maintainer relayed real client-team pain: the **after-session ritual isn't followed through
even with Claude** (busy sessions just end), and a **Copilot-only team has no triggers** for *any* ritual
(end-of-session log, review, sync). The maintainer reaffirmed the Vision (lightweight / no-code /
agent-is-the-runtime) but said **automatic ritual triggers + the end-of-session log should be
reconsidered**. Chose the **full layered design** option.

**Diagnosis.** The weak point is the **trigger layer**, not the ritual: rituals are agent-self-triggered
conventions; the hook story is opt-in, **per-vendor**, **not installed**, and **doesn't travel into
targets** (proven this week on simple-proxy — the opt-in recipe `optional-ritual-hook.md` is tool-only;
a target's `AGENTS.md` only vaguely says "wire a hook").

**Key insight (the design's spine).** The only **vendor-neutral** trigger substrate is **git + CI**
(everyone commits; everyone runs CI), not vendor hooks (fragmented; some modes have none; don't travel).
And the session log splits into **deterministic capture** (scriptable — auto-stub) + **agent judgment**
(the thoughtful summary — irreducible), mirroring `memory-lint`'s "script counts, agent judges meaning."

**Vision reconciliation (important).** This does **not** break the Vision: a git hook / CI / bundled
script is run by the user's env at their opt-in — the **same category** as the `memory-lint` and
`sync-adapters` scripts already shipped (`no-build-step-agent-run`; the tool runs nothing). The change
is flipping triggers from *opt-in-afterthought-that-doesn't-travel* → *first-class, vendor-neutral,
installed/one-command-activated*.

**Proposed layered model (full design in the doc).** Layer 0: agent-primary + strengthen the
"a task isn't *done* until the log exists" framing in `AGENTS.md`. Layer 1 (new, vendor-neutral): a
committed `.githooks/` (via `core.hooksPath`) + a CI workflow that auto-stub the session log, run
`memory-lint` to warn when review is due, and run the `sync-adapters` script. Layer 2: vendor hooks as
an optional real-time nicety, **with the recipe now traveling into targets**. §6 lists the forks for the
plan-gate (nudge vs auto-stub, advisory vs gate, pre vs post-commit, install/activation, the no-code line).

**Status: awaiting maintainer plan-gate** before any implementation. No `VERSION` change (Design-altitude
doc + blueprint thread only; no installed-behavior change yet). Not committed.
